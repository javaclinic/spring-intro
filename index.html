spring-intro

This simple example illustrates Inversion of Control (IoC) and how to implement this with Spring and with no Spring.

There are two runnable applications, e.g.

  • example.run.MyEmployeeNoSpringApp
  • example.run.MyEmployeeWithSpringApp

Both applications configure and instantiate their own bean factory, e.g.

  • example.nospring.configuration.NoSpringBeanFactory (no Spring, our own implementation)
  • org.springframework.beans.factory.BeanFactory (Spring-provided interface) with org.springframework.context.support.ClassPathXmlApplicationContext (Spring-provided implementation)

Once the applications configure and instantiate bean factories, the operations are the same for both application versions. The applications ask for an instance of example.services.EmployeeService type of bean, called myservice. Once they get hold of a service instance, they call the business service method, e.g. example.services.EmployeeService.countEmployees().

This example application illustrates the purpose of Spring's BeanFactory IoC container. We could achieve the same thing with our "own factory", but we would need to code it every time for a new project. Spring's BeanFactory and its derivations (e.g. ApplicationContext, etc.) provide powerful way to wire your application components (i.e. beans).

We presented the most traditional way (i.e. "old way") of configuring Spring container, by using an XML configuration file. There are other ways to configure and wire your components, e.g. autowiring, annotations, no-xml configuration, etc.
